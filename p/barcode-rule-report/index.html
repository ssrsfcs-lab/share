어디서 죽는가 — 스캔 한 번의 여정

BoxHero 자동생성 2079697731717 (20접두 13자리) 기준

1
BoxHero → KPOS 등록 : 정상

sangmast에 20접두 바코드 102개가 이미 등록돼 있음. 등록 단계 문제 아님.

2
라벨 출력 · 스캐너 판독 : 정상

체크디지트 유효 → 스캐너는 잘 읽음. EAN-13이든 Code128이든 스캐너는 그림을 벗겨내고 같은 13자리 숫자 문자열만 키보드 입력으로 POS에 전달. 출력 형식은 무관.

✕
매장 POS 입력 파서 : 여기서 사망

Setup.INI의 Code20=5digit + WeightCodeUse=1 → "20으로 시작하는 13자리"는 상품바코드가 아니라 저울(중량) 바코드로 해석. 13자리 전체로 DB를 찾는 게 아니라 앞자리를 상품코드 5자리로 자르고 뒷부분을 중량/가격으로 읽음 → 그런 상품은 없음 → 품목조회 실패.

핵심: 바코드가 DB에 멀쩡히 등록돼 있어도 매칭이 시도조차 되지 않습니다. 문제는 등록·스캐너·출력형식이 아니라 바코드 값 자체(20접두)입니다.
